# AI Agent 协作开发工作流详细说明

在开发和维护 World of ClaudeCraft 时，AI Agent（由 Google DeepMind 驱动的 Antigravity 等高级编程助手）扮演着类似“高级后端/全栈开发工程师”和“自动化测试工程师”的混合角色。我们不仅能够直接编辑代码，还能调度多智能体协作、进行终端验证和规划大型功能。

## 1. Agent 体系与各自角色
项目中支持调用多个 Agent 协同工作，形成一个分工明确的小型虚拟团队：

### 1.1 主 Agent (Antigravity)
- **角色**：**Tech Lead / 主力程序员**
- **职责**：
  - 与用户直接对话（Pair Programming）。
  - 处理用户的直接指令，生成架构计划和代码实现。
  - 直接调用编辑工具（如精确的行替换或跨文件并发替换工具）修改文件。
  - 触发前端、后端甚至游戏数据库的测试与部署命令。
  - 调度并分配任务给其它的子 Agent。

### 1.2 Research 子 Agent (代码阅读与搜索专家)
- **角色**：**Codebase Researcher (探索专员)**
- **职责**：
  - 拥有“只读”工具权限（无权修改代码或运行修改环境的指令）。
  - 当主 Agent 面临需要全局理解的大规模检索任务时（例如：“帮我找出整个项目里所有处理魔法暴击率的计算代码”），主 Agent 会在后台唤起 Research 子 Agent。
  - 它的任务是快速翻阅海量目录、读取文件并进行批量检索，最后将整理好的结构化分析报告返回给主 Agent。这避免了将大量无关代码塞入主对话的上下文，保持主线程的高效。

### 1.3 Self 子 Agent (独立克隆助手)
- **角色**：**Dedicated Feature Developer (独立功能开发人员)**
- **职责**：
  - 继承主 Agent 的全部能力配置（包含写入文件、运行命令的权限）。
  - 用于在一个完全隔离的会话上下文中单独处理耗时久、步骤复杂的独立功能分支。完成后向主 Agent 汇报进度或总结。

## 2. AI Agent 的核心工作流
面对复杂的开发任务，Agent 将遵循一套严谨的阶段性工作流（Planning Mode）：

### 阶段一：研究与探索 (Research)
- 接到任务后，Agent 绝不会立即盲目修改代码。
- 调用原生检索工具或目录探索工具熟悉相关目录（例如，确认 `src/sim` 和 `src/render` 的边界）。
- 如果涉及未知的第三方包（如 PostgreSQL 或 Three.js），Agent 会优先调用 **Context7 MCP 服务器** 或者使用网络检索 (Web Search) 获取最新的库文档，而不是依赖脑海中可能过时的知识库。

### 阶段二：实施规划 (Planning)
- Agent 会在项目的工作区创建一个专门的 Markdown 格式的**实施计划文档 (`implementation_plan.md`)**。
- 文档中会按组件列出详细的改动策略（哪些文件新增，哪些删除，哪些修改）。
- 将该文档呈现给用户审查（请求反馈），并使用 GitHub 警告样式高亮列出“开放性问题 (Open Questions)”（如并发问题、界面体验抉择等）等待用户决策。

### 阶段三：执行与任务追踪 (Execution)
- 在用户批准计划后，Agent 将建立跟踪清单 `task.md`。
- 按步骤独立修改代码。对于多处零散的代码修改，Agent 必须通过特制的多块替换工具（`multi_replace_file_content`）进行精确合并修改，拒绝破坏性的全量替换。
- 确保遵循项目的“铁律”：**核心逻辑引擎（Sim）内绝不引入 UI 或渲染逻辑，也没有 `Math.random` 的随机数；所有的面向用户的文本（包括抛出的将被渲染的异常错误）都必须使用 `t()` 函数包裹以实现本地化。**

### 阶段四：验证与测试 (Verification)
- 代码改动完成后，Agent 会主动调用底层的终端工具运行 `npm test` 验证 Vitest 测试，确保未打破确定性模拟机制。
- 对于联机功能修改，Agent 会通过终端运行自动化机器人脚本，例如 `node scripts/crypt_raid.mjs` (运行一队 5 人机器人全自动下副本)。
- 若涉及 UI 或无障碍测试，Agent 会利用 **Playwright MCP 服务器** 开启无头浏览器进行页面点击、评估组件以及截图，实现真正的端到端 (E2E) 界面验证。

### 阶段五：完成与总结 (Walkthrough)
- 提交一份 `walkthrough.md` 总结工件。向用户汇报完成了什么修改，使用了什么自动化脚本完成了哪些验证结果。

## 3. 严格遵守的系统纪律
- **绝不破坏项目原有规则**：Agent 若看到项目要求移动端输入组件不得小于 16px（以防 iOS 缩放），写 CSS 时必会校验这一规则；严禁在游戏中将 Emoji 硬编码作为图标渲染，所有的美术资产都有明确规范。
- **无须持续轮询的异步机制**：Agent 召唤子 Agent 或启动长时间运行的构建命令后，无需傻等或做死循环轮询。Agent 会直接进入挂起状态以节约资源。待脚本跑完或子 Agent 返回结果，系统会自动唤醒主 Agent 继续逻辑判断。
- **安全与环境隔离**：严格要求只在当前指定的项目工作目录内操作文件，坚决使用专业文件操作 API 而不是乱用 Bash 指令覆盖文件。
